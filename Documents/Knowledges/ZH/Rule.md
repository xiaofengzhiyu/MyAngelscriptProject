# 知识文档写作建议

> 这不是强制规范，而是一组帮助写出好文档的建议。
> 核心目标：**让读者看完后能清楚一个系统是什么、怎么运转、为什么这样设计。**

---

## 文档结构

一篇文档的基本节奏是 **开头定位 → 正文展开 → 结尾收束**，具体怎么编排由主题决定。

### 开头

- 标题下方先用几句话说清"这篇在讲什么"，让读者快速判断要不要继续读
- 如果涉及特定源码文件，在开头附近列出关键文件路径，方便跳转

### 正文

按主题自然组织小节，不限数量。以下是常见的内容方向，按需组合：

- **结构与流程**：整体架构、核心类/数据结构、关键函数实现、生命周期、状态变迁
- **设计意图**：为什么选这个方案、放弃了什么、换来了什么
- **实现细节**：算法拆解、内存布局、性能考量、边界情况处理
- **环境上下文**：线程模型、配置参数、与其他系统的交互、版本间差异

### 结尾

给出几条关键结论——"这个系统最值得记住的点是什么"。

## 代码相关

### 代码不裸贴

每段代码前后都应有文字说明它在流程里的角色。代码与解释交织出现，像对话一样推进，避免"前半篇文字 + 后半篇代码"。

### 注释写有信息量的内容

```cpp
// ✗ 纯复述，没有信息增量
float MinSampleWeight; // 最小样本权重

// ✓ 解释语义和影响
float MinSampleWeight; // 样本权重下限，低于该值会被抑制，用于降低噪声放大
```

### 用步骤注释划分长函数

函数内部不同阶段用分隔注释标记，既保持代码连续，又让读者看清逻辑分段：

```cpp
void SomeFunction()
{
    // ===== 步骤1：准备输入 =====
    ...

    // ===== 步骤2：核心处理 =====
    ...

    // ===== 步骤3：输出结果 =====
    ...
}
```

### 用 ★ 标记核心调用

在调用链或代码中，用 `★` 标记最关键的那一行，让读者一眼定位。

### 允许有标注的省略

长函数可以省略非核心分支，但要说明省略了什么：

```cpp
    // ... 此处省略 error handling 分支，不影响主流程理解
    //     完整实现见 XXX.cpp L200-L250
```

## ASCII 图示

图放在对应概念首次出现的位置附近，放在裸代码块（无语言标记）中。

### 中英文处理

中文字符是双宽度，在框图中会导致排版错乱。建议：

- **树形图**：节点名英文，行尾 `//` 注释写中文
- **框图**：框内全英文，图后附术语说明
- **流程图**：步骤编号 + 英文关键词，行尾 `//` 注释写中文

### 常用图示类型

**树形结构**（类成员、数据关系、继承层次）：

```
FSceneRenderer
├─ Views[]                              // 视图数组
├─ VisibleLightInfos[]                  // 光源信息
└─ Scene*                               // 场景指针
```

**调用链**（用 ★ 标记核心函数，[Phase N] 标注阶段）：

```
Render()
└─ InitViews()
    ├─ [Phase 1] CreatePackets()
    ├─ [Phase 2] SetupDependencies()
    └─ [Phase 3] ★ FrustumCull()       // 视锥剔除
```

**生命周期**（标注触发函数和状态变迁）：

```
Resource Lifecycle
├─ [1] Create    — AllocateResource()    // Uninitialized
├─ [2] Init      — Resource::Init()      // -> Ready
├─ [3] Active    — Tick() drives         // Ready (can suspend)
├─ [4] PendKill  — ★ BeginDestroy()      // -> PendingKill
└─ [5] Destroy   — ★ FinishDestroy()     // -> Freed
```

## 标题下方的快速说明

每个核心类/函数/流程的标题下方，先用几条短列表给读者一个快速认知，再展开细节。写什么取决于读者最想知道什么：

- **类/结构体**：是什么、谁创建、核心职责
- **函数**：做什么、谁调用、关键步骤、结果给谁
- **流程**：目的、触发时机、输入输出、约束条件

把握两点：开头一句话说清"是什么"；后面的条目按需挑选，不必凑固定数量。

---

## 语言与风格

- 正文和代码注释用**中文**，代码和 UE 术语保持**英文原文**
- 首次出现的英文术语附中文翻译，如 "FrustumCull（视锥剔除）"
- 以码说话，文字叙述简洁，不做过多铺垫
- 先给全景再钻细节——涉及多个类/函数时，先交代整体关系，再逐个展开

---

## 自检参考

写完后可以快速过一遍：

- 每段代码都有解释吗？有没有裸贴的代码块？
- 读者能看懂调用关系吗？（谁调用、参数从哪来、结果给谁）
- 图和表是在帮助理解，还是在凑数？
- 结构体字段是按组解释了语义，还是裸列了一堆？
- 文末有结论吗？读者看完能记住什么？
