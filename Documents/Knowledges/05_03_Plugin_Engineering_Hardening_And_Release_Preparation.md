# 插件工程化硬化与发布准备

> **所属模块**: 工程治理与演进路线 → Plugin Hardening / Release Readiness
> **关键源码**: `Documents/Plans/Plan_PluginEngineeringHardening.md`, `Documents/Guides/Build.md`, `Documents/Guides/Test.md`, `Documents/Rules/GitCommitRule.md`

这一节关注的不是脚本系统本身怎么工作，而是作为一个可复用插件，当前仓库还需要哪些工程化硬化与资料配套。最核心的判断是：**发布准备并不只等于能编译通过，它还意味着构建入口、测试入口、文档主线和治理附录要同时收束。**

## 5.3.1 可复用插件的工程硬化基线

- 插件必须有稳定的模块边界、构建入口、测试入口和最小使用说明
- `Plan_PluginEngineeringHardening.md` 把这件事从“项目开发便利性”提升为“插件交付可用性”问题
- 因此工程硬化基线至少包括：可构建、可测试、可追踪、可比较、可维护

## 5.3.2 构建 / 测试 / 发布资料如何与架构文档配套

- `Build.md`、`Test.md` 解决怎么跑；架构文档解决系统怎么组织；两者不能脱节
- 当前 `Documents/Knowledges/` 系列要补的，就是让“为什么这样组织”与“怎么验证这些组织”互相对齐
- 发布资料因此不只是操作手册，还要能和架构主线互相引用

## 5.3.3 哪些文档属于架构主线，哪些属于治理附录

- 架构主线文档：解释插件边界、运行时主链、生成链、支撑子系统和测试架构
- 治理附录：技术债地图、外部参考吸收路线、构建/测试/提交规则、附录索引表
- 这条分界线很重要，因为如果把治理材料塞进主线正文，架构阅读顺序就会被流程细节淹没

## 当前发布准备最值得记住的点

- 发布准备是一套文档与工程能力共同收束的过程，不是单独的打包步骤
- 构建/测试指南需要和架构主线配套，才能支持可复用插件交付
- 架构正文和治理附录必须分层，否则读者会在入口阶段被过量流程信息打断

## 小结

- 插件工程化硬化的目标是让 `Plugins/Angelscript` 具备清晰、可验证、可交付的外部入口
- 构建、测试、发布资料与架构主线必须形成闭环
- 治理附录不应取代架构正文，但它们是发布准备不可缺少的配套层
