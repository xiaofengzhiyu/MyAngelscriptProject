## 8. 结论

### 8.1 主干现状总结

当前 `AngelscriptProject` 主干已进入**成熟度阶段**，核心能力已落地：

- ✅ **运行时稳定**: 三模块结构、123 Binds、DebugServer V2、CodeCoverage、StaticJIT、24 类状态导出
- ✅ **测试体系化**: 480 自动化测试、主题化组织、引擎隔离机制
- ✅ **编辑器完整**: 菜单扩展、Source Navigation、BlueprintImpact Commandlet

但**对外交付基线严重缺失**，导致仓库仍处于"内部研发可用、外部消费不可行"状态：

- ❌ **入口缺失**: README=NULL、无 CI、插件元数据 URL 全空
- ❌ **上手困难**: 仅 1 个 .as 示例，远低于 Hazelight 的 26 个
- ⚠️ **功能缺口**: Script Subsystem（World/GameInstance）编译失败、7 个已知测试失败

### 8.2 优先级判断

按 `Plan_StatusPriorityRoadmap.md` 的 Phase 依赖关系：

```
Phase 0（口径与 owner）
  -> Phase 1（已知阻塞项 + 交付基线）
    -> Phase 2（上手资产 + 工作流入口）
      -> Phase 3（功能 parity 与验证闭环）
        -> Phase 4（AS 2.38 选择性迁移 + 长期架构）
```

**当前应优先执行**:
1. **P0**: 固定现状口径（三套数字、结构事实）
2. **P1.3**: 补齐插件对外交付基线（README、CI、BuildPlugin）— **最高优先级**
3. **P1.2**: 收口 Script Subsystem — **高优先级**
4. **P1.1**: 清掉已知测试失败 — **高优先级**
5. **P2.1**: 建立示例脚本基线 — **中优先级**

**应延后执行**:
- P3+ 功能扩面（Network/RPC、GAS/EnhancedInput 专项验证）— 待 P1/P2 稳定后
- P4 长期架构（去全局化、2.38 迁移）— 待交付基线稳定后

### 8.3 核心建议

1. **立即补齐对外交付基线**: 当前最阻塞外部可用性的不是功能缺失，而是"找不到入口"。README、CI、BuildPlugin 必须优先于新功能开发。

2. **区分引擎侧与插件侧差距**: 凡涉及 UHT、Blueprint 编译链、引擎对象生命周期改造的能力，必须先回 `Plan_HazelightCapabilityGap.md` 判断边界，避免把引擎补丁误写成插件待办。

3. **固定三套数字口径**: 后续所有 status/roadmap/closeout 文档必须显式注明所用数字属于哪一类基线（已编目/源码扫描/live full-suite），禁止混写。

4. **把负例测试转为正例**: `WorldSubsystem`/`GameInstanceSubsystem` 的编译失败不应长期作为"预期行为"，应尽快推进到可用状态。

5. **先验证再宣称**: 兼容矩阵、支持声明、发布承诺都必须绑定已有测试/构建/打包证据，不能把"希望支持"写成"当前已支持"。
