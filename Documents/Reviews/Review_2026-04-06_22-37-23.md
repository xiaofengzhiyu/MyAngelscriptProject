## 4. 关键问题

### 问题 1：运行时绑定路径混入场景型调试探针，生产路径与临时诊断逻辑没有完全分层

- **Problem**：`Bind_UObject.cpp` 的 `opCast` 绑定逻辑内嵌了针对特定类名和类型名的诊断分支，已经不是通用引擎绑定所必需的最小逻辑。
- **Affected Scope**：`Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_UObject.cpp`
- **Evidence**：`Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_UObject.cpp:140` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_UObject.cpp:166` 通过 `ScenarioInterfaceCastSuccess` 和 `DamageableCast` 条件触发 `UE_LOG`；`Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_UObject.cpp:192` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_UObject.cpp:201` 在接口分支继续输出调试日志。
- **Risk Level**：中
- **Recommended Action**：把这类问题定位日志迁移到受控调试开关、专用测试辅助层或仅测试构建可见的 instrumentation，不要把场景名称判断长期固化在基础绑定路径里。

### 问题 2：核心绑定头文件仍保留明显的临时代码边界，说明主干里存在尚未收口的公共维护面

- **Problem**：`AngelscriptBinds.h` 中有大段 `//WILL-EDIT` / `//END WILL-EDIT` 包裹的逻辑，而且这些逻辑位于核心绑定公共头文件内部，不是局部私有实验代码。
- **Affected Scope**：`Plugins/Angelscript/Source/AngelscriptRuntime/Core/AngelscriptBinds.h`
- **Evidence**：`Plugins/Angelscript/Source/AngelscriptRuntime/Core/AngelscriptBinds.h:495` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/Core/AngelscriptBinds.h:604` 包含 `AddFunctionEntry`、`SkipFunctionEntry`、`AddSkipEntry`、`SaveBindModules`、`LoadBindModules` 等逻辑，并带有 `//WILL-EDIT` 标记与 `//TO-DO` 注释。
- **Risk Level**：中
- **Recommended Action**：把“待收口”的绑定注册/持久化逻辑从公共头文件中剥离到 `.cpp` 或受限内部接口，并在主干中清除 `WILL-EDIT` 这种会误导维护者的临时标识，改为可追踪的问题单或明确注释。

### 问题 3：运行时代码直接依赖 AngelScript 私有内部头，未来 fork 演进会把维护面推向第三方实现细节

- **Problem**：主干并不只是在 vendored third-party 中维护补丁，运行时绑定和调试代码本身也直接 `#include "source/as_*.h"`，等于把插件逻辑耦合到了 AngelScript 的内部实现层，而不是仅依赖稳定公共桥接头。
- **Affected Scope**：`Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_Helpers.h`、`Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_FAngelscriptDelegateWithPayload.cpp`、`Plugins/Angelscript/Source/AngelscriptRuntime/Debugging/AngelscriptDebugServer.cpp`
- **Evidence**：`Plugins/Angelscript/Source/AngelscriptRuntime/AngelscriptRuntime.Build.cs:20` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/AngelscriptRuntime.Build.cs:22` 直接把 `ThirdParty/angelscript/source` 暴露到 `PublicIncludePaths`；随后 `Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_Helpers.h:11` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_Helpers.h:15` 直接包含 `source/as_generic.h`、`source/as_context.h`、`source/as_scriptengine.h`、`source/as_scriptfunction.h`、`source/as_objecttype.h`；`Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_FAngelscriptDelegateWithPayload.cpp:8` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/Binds/Bind_FAngelscriptDelegateWithPayload.cpp:10` 继续依赖 `source/as_context.h` / `source/as_datatype.h` / `source/as_scriptengine.h`；`Plugins/Angelscript/Source/AngelscriptRuntime/Debugging/AngelscriptDebugServer.cpp:23` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/Debugging/AngelscriptDebugServer.cpp:27` 在调试服务器中直接拉入多个 `source/as_*` 内部头。
- **Risk Level**：高
- **Recommended Action**：建立一个受控的 AngelScript-internals 适配边界，把对 `source/as_*.h` 的访问压缩到极少数桥接文件；其余运行时/绑定层通过本项目稳定包装接口访问，避免 2.38+ 选择性迁移时把升级成本扩散到大量插件源文件。

### 问题 4：测试总量很高，但覆盖形状并不均衡，若按运行时子目录看仍存在明显空白带

- **Problem**：当前主干已经有大规模自动化测试定义，但“总量大”不等于“关键子系统都有明确回归入口”。从目录结构与关键字扫描看，`StaticJIT` 和 `FunctionLibraries` 仍缺少显式的 dedicated tests 锚点。
- **Affected Scope**：`Plugins/Angelscript/Source/AngelscriptRuntime/StaticJIT/`、`Plugins/Angelscript/Source/AngelscriptRuntime/FunctionLibraries/`、`Plugins/Angelscript/Source/AngelscriptTest/`
- **Evidence**：`Plugins/Angelscript/Source/AngelscriptRuntime/StaticJIT/` 目录下已有 `AngelscriptStaticJIT.cpp`、`PrecompiledData.cpp`、`StaticJITBinds.cpp`、`StaticJITHeader.cpp` 等多份实现文件；`Plugins/Angelscript/Source/AngelscriptRuntime/FunctionLibraries/` 目录下已有 `AngelscriptScriptLibrary.cpp` 及大量 `*Library.h` / `*Statics.h` 入口；但对 `Plugins/Angelscript/Source/AngelscriptTest/` 的关键字扫描未命中 `StaticJIT`、`FunctionLibraries` 或 `AngelscriptScriptLibrary`，见 `Plugins/Angelscript/Source/AngelscriptTest/Internals/AngelscriptTokenizerTests.cpp:18` 这类能明确锚定内部组件的测试风格，对照下更能说明上述两块当前缺少同等清晰的测试落点。
- **Risk Level**：高
- **Recommended Action**：不要只继续累计全局测试数；应围绕 `StaticJIT` 与 `FunctionLibraries` 建立能直达模块职责的测试命名与前缀，让审计者能从测试树直接看出这些能力是否被覆盖。

## 5. 风险与阻塞

- **高风险**：第三方 fork 维护压力已经是现实成本，而不是未来假设。仓库内部 grep 到 `73` 个 `[UE++]` 标记分布在 `32` 个 vendored AngelScript 文件中；例如 `Plugins/Angelscript/Source/AngelscriptRuntime/ThirdParty/angelscript/source/as_string.h:41` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/ThirdParty/angelscript/source/as_string.h:43`、`Plugins/Angelscript/Source/AngelscriptRuntime/ThirdParty/angelscript/source/as_builder.cpp:50` 到 `Plugins/Angelscript/Source/AngelscriptRuntime/ThirdParty/angelscript/source/as_builder.cpp:52` 已明确标注本地改动。这里的风险并不是“没有修改标记”，而是插件代码本身已经越过公共边界直接依赖 `source/as_*.h`，使 fork 差异同时分布在 vendored code 与插件运行时代码两侧。
- **中风险**：当前主干确实已经有大规模 dump/debug/test 资产，但资产规模越大，越需要把“正式 API / 正式 runner / 临时诊断代码”分层清楚。否则能力越多，越容易在后续收敛、升级或对外复用时产生误判。
- **高风险**：测试资产的“体量感”可能掩盖结构性盲区。主干里确实有 `492` 个以上的测试宏命中，但从运行时目录到测试目录的映射看，`StaticJIT` 与 `FunctionLibraries` 这类直接影响交付能力的子系统仍没有清晰的 dedicated coverage 锚点；这会让后续人员高估现有回归闭环的实际宽度。
- **推断**：结合外部 AngelScript 许可要求“altered source versions must be plainly marked as such”与本仓库现有的 `[UE++]` 约定，可以判断团队已经意识到 vendored 修改追踪的重要性；但只要运行时层继续直接吃第三方内部头，后续 selective migration 的风险就不会只停留在 `ThirdParty/angelscript/` 目录内部。

## 6. 文档与测试缺口

- **已验证事实**：构建与测试 runner 文档已经收敛到官方脚本入口，`Documents/Guides/Build.md:5` 与 `Documents/Guides/Test.md:5` 都明确给出唯一标准入口，这一块比很多同体量插件工程更稳。
- **当前缺口**：与 runner 指南相比，运行时内部调试探针、`WILL-EDIT` 临时代码边界、以及运行时直接依赖 `source/as_*.h` 的约束，尚未看到同等强度的工程规则落点。如果这些边界被认为是“当前阶段可接受的特例”，仓库需要把理由、范围和后续收口条件写进贡献规则或迁移说明，而不是只留在代码里让维护者自行猜测。
- **当前缺口**：测试文档强调 runner、group 和 suite 入口，但从源码树本身看，审计者仍难以直接把 `StaticJIT`、`FunctionLibraries` 之类的运行时职责映射到清晰的测试前缀或测试目录。现有测试规模很大，但“哪些运行时子系统已被覆盖到足以支撑主干决策”仍不够一眼可见。

## 7. 建议的下一步动作

- 第一优先级：为 AngelScript 内部头访问建立白名单式适配边界，先收缩 `source/as_*.h` 的直接使用点，再谈更大范围的 2.38 选择性吸收。
- 第一优先级：为 `StaticJIT` 与 `FunctionLibraries` 增加可直接指认模块职责的测试入口，不要让这两块继续淹没在“大量已有测试”带来的覆盖错觉里。
- 第一优先级：把 `Bind_UObject.cpp` 中与特定场景名绑定的诊断逻辑移到可控 instrumentation 层，避免继续污染基础绑定路径。
- 第二优先级：清理 `AngelscriptBinds.h` 中的 `WILL-EDIT` / `TO-DO` 存活区，把公共头文件重新收敛到稳定 API 与声明职责。
- 第二优先级：把 vendored fork 约定显式写进贡献/迁移文档，说明 `[UE++]` 标记规则、哪些文件允许直接接触 AngelScript internals、哪些模块必须通过包装层访问。

## 8. 结论

- 当前主干不是“能力未落地”，而是“能力已经很多，边界治理开始跟不上体量”。三模块骨架、绑定规模、自动化测试规模、DebugServer V2、DumpAll 入口与 BlueprintImpact commandlet 都说明主干已有成熟资产。
- 真正需要优先处理的不是再证明一次“项目很大”，而是降低未来维护与迁移成本：把运行时代码从第三方内部实现细节上解耦、把临时诊断逻辑从主路径清走、把公共头文件里的临时区段收口，并把 `StaticJIT` / `FunctionLibraries` 这类关键运行时子系统补上可见的 dedicated coverage。这些问题如果继续累积，会直接削弱 `Plugins/Angelscript` 作为可复用插件交付物的可信度。
