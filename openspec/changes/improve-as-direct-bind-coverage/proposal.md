## Why

`AngelscriptUHTTool` 现在只把直绑函数表 emit 到 `AngelscriptRuntime` 一个模块。`AngelscriptHeaderSignatureResolver.HasLinkableExport` 因此把所有"非 `AngelscriptRuntime` 模块、且 C++ 实现没有 `<MODULE>_API` / 非 inline / 非 constexpr"的 `BlueprintCallable` 函数判为 `unexported-symbol` → 退化成 `ERASE_NO_FUNCTION()` stub,运行时被迫走 `BlueprintCallableReflectiveFallback`(虽有 `FReflectiveParamCache`,作者注释自陈仍比直绑慢 3-6×)。`AS_FunctionTable_SkippedReasonSummary.csv` 中 `unexported-symbol` 行就是这部分今天本可以直绑、却被链接边界挡住的函数。Verse 在引擎内已用 UHT 把 `FVerseCallableThunk` emit 进函数所在模块自己的 OutputDirectory、再通过 CoreUObject 中的 `RegisterVerseCallableThunks` 中央 API 落到 UClass 元数据上(每个引擎模块本就 link CoreUObject,无需新增依赖),AS 完全可以同形复用 — **但因为我们不能修改引擎源码,会合点改用 UE 现成的 `IModularFeatures`(同样在 Core)**。

## What Changes

- **Scope correction**: the safe automatic subset explicitly excludes static `ScriptMethod` functions and class-level `ScriptMixin` projections. Those forms turn a C++ parameter into script `this`; the current raw thunk bridge passes only explicit generic-call argument slots and does not yet model implicit script-this injection. They remain on fallback/stub behavior or `cross-module-unsupported-signature` diagnostics for a follow-up change.

- **AngelscriptUHTTool 输出形态**:`AngelscriptFunctionTableExporter` 保留 `ModuleName="AngelscriptRuntime"`(UE UHT 插件 exporter 必须指定插件模块;且 plugin factory 下 `factory.MakePath(...)` 会回落到插件模块 OutputDirectory),但 generator 改为基于目标 `UhtModule.Module.OutputDirectory` 显式构造输出路径并 `CommitOutput(...)`。每个目标模块在自己的 OutputDirectory 下额外 emit 一份 `AS_FunctionTable_<Module>_CrossModule_*.cpp`(沿用现有 `AS_FunctionTable_*.cpp` CppFilters 前缀,避免被引擎 `CodeGen` exporter `CullOutput` 误删;Day-0 probe 负责证明该绝对路径输出会被 UBT 纳编到目标模块)。
- **跨模块中央会合点 — `IModularFeatures`(Core)**:每个引擎模块 emit 的 cpp 在 DLL 加载时通过静态构造把一个 POD-payload feature 注册到 `IModularFeatures::Get()`,字符串 ID 为 `"AngelscriptCrossModuleBindings"`。AS Runtime 在 `EOrder::Late + 60` 阶段调用 `GetModularFeatureImplementations(...)` 一次性拉取,**完全不 link 任何引擎模块**(`IModularFeatures` 在 Core,所有引擎模块与 AngelscriptRuntime 本就依赖 Core)。后到的模块通过 `OnModularFeatureRegistered` 委托补注册。
- **跨模块 ABI(POD payload + reinterpret_cast 反向 cast)**:新增公共头 `Plugins/Angelscript/Source/AngelscriptRuntime/Public/UHT/AngelscriptCrossModuleBindings.h`,定义 `FAngelscriptCrossModuleEntry { const TCHAR* ClassName; const TCHAR* FunctionName; void(*Thunk)(UObject* Self, void** Args, void* Ret); uint16 ArgCount; uint16 RetSize; uint32 Flags; }` 与 layout-matching 的 `FAngelscriptCrossModuleFeatureReader`、`LayoutVersionExpected` 常量、双端 `static_assert(sizeof)` 校验。UE 5.7 的 `IModularFeature` 当前是空接口,reader 不包含 vtable padding;若未来 Core 接口变成多态,Day-0 probe / ABI static_assert 必须先失败。**该头只被 AS Runtime 自己 include**;引擎模块 emit cpp 的同型 layout 由 generator 通过模板字符串保证两端一致,不反向依赖 AS Runtime 的 Public 头。
- **Thunk 签名为 raw 形态,不暴露 AS SDK**:emit cpp 的 thunk 形态是 `void(*)(UObject* Self, void** Args, void* Ret)`,**只 include `Features/IModularFeatures.h` 与目标类头**,不 include `angelscript.h` 也不 include 任何 AS Runtime 头。`asIScriptGeneric*` ↔ raw 缓冲区的桥接由 AS Runtime 端一个 *单一通用 generic hook* `void(*)(asIScriptGeneric*)` 完成 — 它从 generic 拿 Self/Args/Ret 放进 raw buffer,再调 entry 的 thunk;所有 cross-module entry 复用同一个 hook,通过 AS user-data 携带 entry 指针。
- **AngelscriptRuntime 端注入**:新增 `Bind_CrossModuleDirect.cpp`,`EOrder::Late + 60` 阶段执行:`IModularFeatures::Get().GetModularFeatureImplementations(FName("AngelscriptCrossModuleBindings"))` → 逐个 `reinterpret_cast` 到 reader struct → 校验 `LayoutVersion` magic → 通用 generic hook 注册到 AS Engine(`asCALL_GENERIC`)→ 写回 `FAngelscriptBinds::ClassFuncMaps` 空槽。同时 `OnModularFeatureRegistered` 处理后到模块。
- **HasLinkableExport 收敛**:不再因为"跨模块 + 无 API 宏"直接判 `unexported-symbol`。cross-module resolver 会先尝试解析并通过安全签名 gate；能安全 emit 的候选进入 `CrossModule` 类目，复杂但可识别的签名记录为 `cross-module-unsupported-signature`，目标模块不可用记录为 `target-module-disabled`，RPC/Net 记录为 `rpc-net-function`。
- **RPC / Net 函数继续走反射 fallback**:`FunctionFlags & (FUNC_Net | FUNC_NetServer | FUNC_NetClient | FUNC_NetMulticast)` 命中的 UFunction **绝不** emit cross-module direct bind。原因:`UFunction::Invoke` / `ProcessEvent` 内部按 `FUNC_Net` 路由 RPC,raw thunk 直接 `static_cast<T*>(Self)->Func()` 会绕过路由,把"应 marshal 到对端"的语义变成"本地直接执行函数体",**破坏网络复制**。RPC 函数继续走 `BlueprintCallableReflectiveFallback`,行为不变。
- **安全签名子集的 thunk 解包契约**:本 change 自动 emit 的范围收敛到不会改变 BPVM 语义、无需额外写回协议的签名：返回 `void`、bool/numeric/enum/struct、`FString`、`FName`、`FText`、`UObject*`；参数为 bool/numeric/enum/struct、`FString`、`FName`、`FText`、`UObject*`、`UClass*`、soft object、weak object。明确排除 out-param、WorldContext、ref-return、static arrays 与 `TArray/TSet/TMap` 容器；这些继续走 fallback 或进入 `cross-module-unsupported-signature` 诊断，后续单独扩展。
- **Feature 实例 constructor-only**:`IModularFeature` 在 UE 5.7 当前是空接口,但 feature payload 不走 brace-aggregate-init 作为长期约束。所有 emit cpp 与 AS Runtime 公共头中 `FAngelscriptCrossModuleFeature` 必须带显式 ctor,实例化用 `static FFeature GFeature(GTable, UE_ARRAY_COUNT(GTable), TEXT("Engine"), 0xA5C0DE01u);`。
- **LayoutVersion bump 契约**:任何对 `FAngelscriptCrossModuleEntry` POD 段、`FAngelscriptCrossModuleFeature` POD 段的 add/remove/reorder/widen/narrow 字段 **MUST** bump `LayoutVersionExpected`。值由 `Plugins/Angelscript/Source/AngelscriptUHTTool/cross-module-layout-version.txt`(单一 token 文件)托管,generator C# 与 AS Runtime 公共头都从该文件读,**两端不可能再漂移**。
- **stale shard 清理跨所有 supported module**:现有 `DeleteStaleOutputs` 只 enumerate AngelscriptRuntime 自己的 output dir;新方案下 cross-module shard 散落在 N 个引擎模块下,某函数被加 `NotInAngelscript` 后旧 shard 不删 → UBT 仍编 → 重复定义 link error。`AngelscriptFunctionTableCodeGenerator.DeleteStaleOutputs` 必须扩展到所有 supported module 的 OutputDirectory。
- **同模块多 shard = 多个 feature 实例**:大模块按 `MaxEntriesPerShard = 256` 分片,每个 shard 文件有自己的 anonymous `GFeature` + `GAutoReg`,**同一引擎模块在 IModularFeatures 注册表中会出现多条同名 feature**。AS Runtime 按 feature 实例迭代而非按 ModuleName 去重;ModuleName 字段仅用于诊断/日志。
- **Shutdown 时序安全**:`IModularFeatures` 在 UE 5.7 没有全局 `IsAvailable()` API。`~FAutoReg()` 不依赖不存在的 availability 查询;AS Runtime 与 emit cpp 双方在 `FCoreDelegates::OnPreExit` 设置 shutdown flag 并主动移除委托,`~FAutoReg` 看到 flag 后 no-op,避免 Core 单例析构顺序不确定时再调 `IModularFeatures::Get()`。
- **`OnModularFeatureRegistered` 线程安全**:UE 不保证该委托在 main thread 触发(动态加载的 plugin 可能在 worker thread 调 `RegisterModularFeature`)。AS Runtime 的回调里写 `ClassFuncMaps` 与 AS Engine `BindMethodDirect` 都 *不* 是 thread-safe(参考 `Bind_BlueprintCallable.cpp:324` 注释"AS Engine register half (must run on GameThread)")。回调必须 `AsyncTask(ENamedThreads::GameThread, ...)` 把工作 marshal 到主线程。
- **Modular / Monolithic 设计路径**:机制本身不依赖 PE 导出表；Editor source-build 已作为当前验证目标。Monolithic Shipping 与 Launcher 安装引擎的完整烟测不作为本次 safe-scope 完成门槛，保留为后续 release-hardening 矩阵。Launcher / 不重编引擎用户的目标模块没编出 cross-module shard 时，运行时 `IModularFeatures` 拉空，沿用现有 fallback。
- **三道 ABI 防护**:(a) 编译期 `static_assert(sizeof(FAngelscriptCrossModuleEntry) == ...)` 双端等价;(b) `LayoutVersion` magic 字段不一致即 runtime warn + skip;(c) feature 进 AS Runtime 时核 `Count >= 0`/`Table != nullptr`/`ModuleName != nullptr`,失败 skip,不 crash。
- **Day-0 link probe 作为不可逾越的前置**:Phase 0 第一项任务在 Engine 模块显式输出一个最小 IModularFeatures self-register cpp,验证 (i) 插件 UHT exporter 使用目标模块 `OutputDirectory` 绝对路径 `CommitOutput` 时,UBT 是否纳编该额外 .cpp 到目标模块,(ii) AS Runtime 端能否通过 `GetModularFeatureImplementations` 在 Late+60 拿到。失败即 STOP,不继续后续工程化。

不属于本 change:`UASFunction` 内部 dispatch / StaticJIT / 反射 fallback 内部实现 — 这些已由 `uasfunction-runtime-dispatch-coverage`、`uasfunction-dispatch-matrix-and-jit-paths` 覆盖。
